# Git Workflow (простая стратегия веток)

## 🎯 Цель

Минимальная и понятная схема без перегруза:

- быстро разрабатывать
- не накапливать мусор
- сохранять контроль над стабильными версиями

---

## 📦 Структура веток


main ← стабильные версии (релизы)
dev ← основная разработка
feature/* ← короткоживущие ветки под задачи


---

## 🧠 Принципы работы

### 1. Вся работа через `dev`

```bash
git checkout dev
всегда актуальное состояние проекта
сюда мержатся все изменения
допускаются эксперименты
2. Каждая задача = отдельная ветка
git checkout dev
git checkout -b feature/<название>

Примеры:

feature/hybrid-search
feature/calibre-import
feature/ui
feature/ranking-tuning
3. Завершил задачу → сразу merge и удалить ветку
git checkout dev
git merge feature/<название>
git branch -d feature/<название>
4. main — только для релизов

Не используется для разработки.

Когда готова стабильная версия:

git checkout main
git merge dev
git tag v0.1
📊 Как это выглядит
main ────────●──────────────●──────
              \            /
dev  ─────────●────●────●────●─────
               \    \    \
feature/*       ●    ●    ●   (удаляются после merge)
⚠️ Чего избегать
❌ Не работать напрямую в main
❌ Не держать много feature-веток

Если их больше 2–3 → пора чистить

❌ Не хранить старые ветки

Удалять после merge:

feature/old
backup/*
test/*
💡 Правило в одну строку

Одна задача → одна ветка → merge → удалить

🚀 Применение в проекте

Текущий процесс:

dev

feature/hybrid-search
feature/calibre-import
feature/ui

Работа идёт последовательно:

создаётся feature-ветка
реализуется задача
merge в dev
ветка удаляется
📌 Именование веток

Использовать простой формат:

feature/<что_делаешь>

Примеры:

feature/hybrid-search
feature/ui
feature/calibre-import
✅ Итог
минимальная структура
нет лишних веток
понятная история
контроль через main
скорость разработки через dev

---

Если хочешь, можно следующим шагом добавить туда ещё:

- или чеклист “когда делать релиз”
